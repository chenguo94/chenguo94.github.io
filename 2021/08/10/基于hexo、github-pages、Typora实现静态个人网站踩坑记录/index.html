基于hexo、github pages、Typora实现静态个人网站踩坑记录

前言

​ 实现个人网站的方式有很多种，看完2020 年，如何才能拥有一个个人网站？ - 少数派 (sspai.com)这篇之后，决定使用hexo+github pages完成。WordPress是个很火的选择，但其主页不知道为啥似乎被墙了，不能在上面托管的话就得自己折腾云服务器了，暂时还不想搞这么麻烦的，以后可能会整一个玩玩吧。

​ 接触这一堆玩意儿的第一时间我是有点懵的，所以首先来梳理下这些都是干啥的。

  • Typora：一个在本地使用的markdown编辑器，用来编辑.md文件。
  • Hexo：一个框架，用来把本地的.md文件基于配置和模板渲染成.html，并帮你部署到github上。
  • GitHub Pages：用github库来存放渲染完后的网站文件，感觉可用性比我随便找个不知名的静态空间还是强点，话说回来，1202年了，现在还有那种网页三剑客时代的静态空间么==！
  • Node.js：一个框架，需要它提供的npm来安装hexo，真挺慢的，没魔法建议换源。
  • Git - Downloads ：git是个版本控制程序，github是个网站，我就知道这么多。理论上hexo自己调用这玩意儿就完事了，但很不幸我后面还是折腾了一堆。
  • 实现过程：本地编辑好.md文件（基于typora或者其他的），hexo渲染为静态网页，发布到github pages。我看网上也有人不用前两者直接使用github pages+上传模板建站的，那种可能更适合搞好了不用再改的展示网站吧。

3.github建立repository和pages

真正的坑。。。主要还是由于对github太不熟悉

  1. Repository name 中输入 用户名.github.io，此处必须一字不差，第一次敲完我发现得到的库地址是https://github.com/用户名/用户名.github.io，以为自己多敲了个用户名，顺手给删了，然后就折腾报错半天。

  2. pages现在在库-settings-pages，很多老教程不是的。

  3. 库建好后很多教程会直接开始整ssh连接，生成private key和public key然后上传啥的。个人觉得实在没必要，https方式又不是不能用。。。so，这里的区别就在于，使用ssh方式的话地址用库界面给的git@github.com:xxxx，https的话就是https://github.com/用户名/用户名.github.io.git，说是比ssh要多敲几遍用户名密码，其实支持点浏览器授权，鼠标点两下就好了~这里墙裂推荐新手放弃设置ssh，少个环节少点报错的可能~~
